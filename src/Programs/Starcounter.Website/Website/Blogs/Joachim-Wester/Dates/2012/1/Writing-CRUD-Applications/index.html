Writing CRUD Applications

January 18, 2012 by: Joachim Wester

Applications that Create, Read, Update and Delete are should be so simple, and yet, they create so much headache. Writing a local application for a single user is simple. Writing a scalable multiuser system is hard.

If you architect applications with many users, your bottleneck is ultimately going to boil down to management of shared state.

Static data or data not requiring synchronization/transactions can be replicated cheaply across many small commodity servers. In resent years, many database solutions have come to handle this nicely (especially the newer NoSQL* databases that were designed for horizontal scaling from the get go).

Transactional update of data however is a very thorny problem. If you want the complete theory, you can read about Brewers theorem* (a.k.a. the CAP theorem), the Byzantine Generals' Problem* and the speed of light limitation following from Einsteins theory of relativity*. To save you time, what it all boils down to is that it is simply that it is impossible to scale-out consistent data. You have to give up one of two things; data consistency (correct results) or horizontal scaling*. So if you have ever found it frustrating that Facebook status texts pop in and out of existence before they settle, don't blame the developers.

But even if Facebook updates can allow "fuzzy" updates, there are situations where you need correct data. After all, you don't want two clients to book the same seat on an airplane or have your mismatches in your ERP accounting.

The solution is to scale in. For some web gurus, this might come as a surprise. After all, horizontal scaling is what we want, right? But the reason why SAP is moving to HANAH, a single node RAM database, is not because they missed to notice the possibility on horizontal scaling on cheap commodity servers, it was because they need consistent data. Just as the perpetual machine, it is easy to look at a drawing and think, hmmmâ€¦ this just might work. But latency and the CAP theorem is merciless.

This is why we created Starcounter. We capitalize on the fact that a modern computer is very, very fast. In fact, the 66 MIPS and the xx GFLOPS of a web server is faster than most supercomputers was when the first version of the Apache web server was created**. A modern web server can perform a hundred operations per second for every person living on planet earth every second**. CPU power is not a problem. And a single network card can do 10GbE. This is more than Wikipedia uses globally**. On a single network card.

Starcounter can handle millions of simultaneous users on a single machine with full ACID transactions. You still want to add a few extra commodity servers on these volumes, but that is just to serve static content and to add redundancy.

Using this approach makes allows your code to be clean, simple, beatiful beautiful and focused. Focused on write wonderful user functionality, not on satisfying outdated architectures.

Packets travel at best .66 (optical) and ~.55 (copper) times the speed of light.
