Don’t Give Up Consistency

February 8, 2012 by: Peter Idestam-Almquist

If you have a successful application and it relies upon persistent data then data storage will probably become your bottleneck. There are two main strategies for scaling an application: vertical scaling and horizontal scaling.

Horizontal scaling means distributing the data over several database nodes. According to the CAP theorem you can only have two out of consistency, availability and partioning tolerance. Since horizontal scaling is based on partitioning and availability is required for online applications, you typically give up consistency.

However, when you give up absolute / immediate / global consistency for some weaker relative / eventual / local consistency you move the problem from the database management system to your application developer and/or your end users. When you have a many-user application then transactional conflict will occur. When you relax the consistency requirements of your database system, you either need to resolve the transactional conflicts in your application or propagate the problem of the (temporary) inconsistency all the way to the end users: "Sorry, we have just sold you a book we have already sold to someone else." Sometimes you have to accept inconsistent data, but it is never desirable.

Vertical scaling means getting more processing power of a single database node. This is much easier since you have a centralized database solution where you do not have to give up absolute / immediate / global consistency. Typically you buy more powerful hardware. Today, you can also get a hundred times more power from the same hardware by replacing your traditional relational database system (OldSQL) to a modern database (Starcounter) which better utilies the possibilities of modern hardware. Thus the limitations of vertical scaling has decreased significally.
