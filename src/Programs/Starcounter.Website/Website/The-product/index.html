Get high performance AND consistent data

The data management challenge of today is how to handle:

  • big volumes of data, and

  • huge number of transactions on the data.

When you deal with updatable data, the problem is how to achieve both:

  • high performance, and

  • consistent data.

Your alternatives

OldSQL

- The market dominating old relational databases (OldSQL) can only process tens of thousands database transactions per second (TPS) on a single machine.

+ They support ACID transactions which guarantee consistency.

NoSQL

+ NoSQL databases distribute the data over several machines and in this way can process many more database transactions per second.

- They cannot provide data consistency and the simplicity that it brings. 

NewSQL

+ Starcounter (NewSQL) can process millions of database transactions per second on a single machine.

+ These database transactions are ACID and thus consistency is guaranteed.

 

Starcounter is the only alternative that gives you both high performance (high throughput and low latency) and consistent data.

Starcounter scales to the need of almost any application. You scale a Starcounter installation by adding more RAM and more cores to the server where Starcounter is running.

Starcounter is SUPER FAST!

Starcounter is, in contrast to OldSQL databases, originally designed to have its main storage in RAM, to utilize modern multi-core CPUs with several level of caches, and to minimize overhead. Starcounter also makes use of a new invention we call VMDBMS, which makes it substantially faster than other in-memory high performance databases.

VMDBMS stands for an integration between the application runtime virtual machine (VM) and the database management system (DBMS). As a result of this integration the database data resides all the time in one single place in RAM and is not copied back and forth between the database and the application.

When the application accesses database data, it directly accesses the data in the database's part of RAM. Thus, unlike Entity Framework, nHibernate and other OR-mappers, the objects do not hold any temporary local copy of the data in the application's part of RAM. This means no serialization or deserialization of objects is needed.

Performance

Starcounter performs 500,000 ACID read-only transactions per second and core, and it scales almost linearly on the number of cores.

Starcounter performs 100,000 ACID read-write transactions per second on one core. It does not scale on the number of cores but reaches a maximum of about  300,000 read-write transactions per second on four cores.

SUPER EASY .NET object API

Using Starcounter .NET object API, the interaction between the application and the database becomes super easy. You can write your plain old C# objects (POCO), and all classes inheriting the class Starcounter.Entity will automatically be included in the database schema.

The .NET object API also gives you an easy way to write SQL queries returning true .NET objects. This makes it possible to combine database queries and programming code in an easy and powerful way.
